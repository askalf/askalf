SUBSTRATE & SIGIL

Internal technical documentation for the AI memory infrastructure powering Ask ALF.

Introduction

SUBSTRATE is a model-agnostic memory infrastructure that enables AI systems to learn, remember, and improve over time. Unlike traditional AI assistants that forget everything between sessions, SUBSTRATE provides persistent, context-aware memory that compounds in value.

Design Principles

  • Model-Agnostic — Works with any LLM provider (OpenAI, Anthropic, Google, local models)
  • Multi-Tenant — Isolated memory spaces per user/organization with configurable visibility
  • Sustainable — Reduces API calls through learned responses (shards)
  • Privacy-First — User data is encrypted and never shared across tenants

Core Components

  • Procedural Shards — Learned response patterns that answer questions without calling an LLM
  • Reasoning Traces — Input/output pairs logged for analysis and synthesis
  • Episodes — High-level interaction summaries with lessons learned
  • Knowledge Facts — Extracted subject-predicate-object relationships
  • Working Context — Session state with liquidation for long-term storage

Core Concepts

Visibility Levels

All memory objects have a visibility field controlling access:

LevelDescription
publicVisible to all users (system knowledge)
privateVisible only to owner tenant
orgVisible to organization members

Lifecycle States

Shards progress through lifecycle states based on performance:

StateDescription
candidateNewly created, under evaluation
promotedProven reliable, actively used
deprecatedMarked for removal due to poor performance

Tenant Isolation

Each user has a tenant_id that provides complete data isolation. All queries filter by tenant to ensure users only see their own data plus public system content.

Procedural Shards

Shards are learned response patterns that answer questions without calling an LLM. When a user asks something that matches a shard pattern, the response is returned instantly and for free.

Shard Structure

{
  "id": "uuid",
  "name": "Calculate percentage",
  "patterns": ["what is X% of Y", "calculate X percent of Y"],
  "template": "{{X}}% of {{Y}} is {{result}}",
  "procedure": "const result = (X / 100) * Y; return { result };",
  "confidence": 0.95,
  "lifecycle": "promoted",
  "execution_count": 1847,
  "success_count": 1832,
  "owner_id": "tenant-uuid",
  "visibility": "public"
}

Pattern Matching

Shards use a combination of:

  • Exact matching — Direct string comparison after normalization
  • Semantic matching — Vector similarity using embeddings (threshold: 0.85)
  • Template matching — Pattern extraction with variable slots

Promotion Criteria

Candidates are promoted to active shards when:

  • Success rate > 90% over 10+ executions
  • No user corrections in last 24 hours
  • Confidence score > 0.8

Reasoning Traces

Every LLM interaction is logged as a reasoning trace for analysis and potential synthesis into shards.

Trace Schema

{
  "id": "uuid",
  "input": "What is 15% of 200?",
  "output": "15% of 200 is 30.",
  "pattern_hash": "sha256-hash",
  "tokens_used": 45,
  "execution_ms": 850,
  "synthesized": false,
  "source": "chat",
  "model": "gpt-4o-mini",
  "owner_id": "tenant-uuid"
}

Synthesis Pipeline

Traces are periodically analyzed for synthesis opportunities:

  1. Group similar traces by pattern hash and semantic similarity
  2. Extract common patterns and variable slots
  3. Generate candidate shard with procedure
  4. Validate against historical inputs
  5. Create candidate shard if validation passes

Episodes

Episodes are high-level summaries of multi-turn interactions, capturing the overall context and lessons learned.

Episode Schema

{
  "id": "uuid",
  "type": "problem_solving",
  "summary": "User debugged a React component rendering issue",
  "success": true,
  "valence": 0.8,
  "importance": 0.7,
  "lessons_learned": [
    "Check useEffect dependencies first",
    "Use React DevTools to trace re-renders"
  ],
  "owner_id": "tenant-uuid"
}

Knowledge Facts

Facts are extracted subject-predicate-object relationships that build a user's personal knowledge graph.

Fact Schema

{
  "id": "uuid",
  "subject": "User's main project",
  "predicate": "is",
  "object": "a React dashboard",
  "statement": "The user's main project is a React dashboard",
  "confidence": 0.9,
  "sources": ["trace-uuid-1", "trace-uuid-2"],
  "category": "user_context"
}

Fact Categories

  • user_context — Information about the user
  • domain_knowledge — Technical or subject knowledge
  • preferences — User preferences and styles
  • relationships — Connections between entities

Working Context

Working context captures session state that may be liquidated into long-term memory.

Context States

StatusDescription
rawActive session data, full fidelity
liquidatedCompressed for long-term storage
promotedExtracted to permanent memory (facts/episodes)

Liquidation Process

When a session ends or context grows too large:

  1. Extract key facts and relationships
  2. Summarize interaction episodes
  3. Compress context using LLM summarization
  4. Store compressed version with importance score
  5. Archive or delete raw context based on policy

SIGIL Protocol Overview

SIGIL (Structured Intelligence Gateway Interface Layer) is the protocol for encoding and routing context between SUBSTRATE and LLM providers.

Design Goal: Provide rich context to any LLM without provider lock-in, while preserving privacy and optimizing token usage.

Protocol Layers

  • Encoding Layer — Serializes memory objects into prompt-friendly format
  • Routing Layer — Selects optimal context for the current query
  • Privacy Layer — Filters sensitive information before LLM calls
  • Compression Layer — Reduces token usage through summarization

SIGIL Encoding Format

Memory is encoded as structured blocks that any LLM can understand:

<SUBSTRATE_CONTEXT>
<user_facts>
- User prefers concise code examples
- User works with React and TypeScript
- User's timezone is UTC-5
</user_facts>

<relevant_episodes>
[2024-01-15] Debugged React state issue - key lesson: always check dependency arrays
</relevant_episodes>

<session_context>
Current topic: Building a dashboard component
Previous messages: 3
</session_context>
</SUBSTRATE_CONTEXT>

Token Budget

Context is allocated based on model limits:

Model ClassContext Budget
GPT-4o / Claude Sonnet2000 tokens
GPT-4o-mini / Claude Haiku1000 tokens
Reasoning models (o1, Opus)3000 tokens

Context Routing

The routing layer selects the most relevant context for each query:

Relevance Scoring

  1. Recency — Recent context weighted higher (decay factor: 0.95/hour)
  2. Semantic Similarity — Vector distance to current query
  3. Importance — Pre-computed importance scores
  4. Usage — Previously helpful context boosted

Routing Algorithm

function selectContext(query, budget) {
  const candidates = await getRelevantMemory(query);
  const scored = candidates.map(c => ({
    ...c,
    score: recencyScore(c) * 0.3 +
           similarityScore(c, query) * 0.4 +
           importanceScore(c) * 0.2 +
           usageScore(c) * 0.1
  }));
  return packWithinBudget(scored, budget);
}
